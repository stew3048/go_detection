## 二、YOLO Conf Threshold 不公平的問題

### 發現

使用者觀察到：法一 Mahalanobis 漏了 2 顆黃棋，法二 AutoEncoder 沒有漏，
雖然兩個方法都用了 YOLO，但原因是 **YOLO conf threshold 設定不一樣**。

```
法一推論：CONF = 0.20
法二推論：CONF = 0.10

那 2 顆黃棋的 YOLO conf ≈ 0.12：
  0.12 < 0.20 → 法一過濾掉 → 根本沒截到 crop → FN（漏報）
  0.12 > 0.10 → 法二保留   → 截到 crop → AE 分析 → ALIEN ✅
```

### Conf 的真正含義

使用者原本認為 conf 是「判斷顏色的信心」，討論後釐清：

```
conf = P(這裡有物件) × P(這個類別正確 | 有物件)

黑棋（YOLO 學過）：
  P(有物件) = 0.95  × P(black) = 0.98  → conf = 0.931

黃棋（YOLO 沒學過）：
  P(有物件) = 0.15  × P(black) = 0.80  → conf = 0.12
                ↑
    形狀像棋子但顏色怪，整體不確定
```

conf 同時包含「位置確定程度」和「類別確定程度」，不只是顏色。

### 使用者的論點

> 「兩個方法對 YOLO 的用途完全一樣（定位），所以 threshold 應該統一。」

這個論點正確。修正後統一使用 `CONF = 0.10` 重跑評估。

### 修正後結果（公平比較）

| 指標 | Mahalanobis | AutoEncoder |
|------|:-----------:|:-----------:|
| Precision | **1.000** | 0.952 |
| Recall | **1.000** | **1.000** |
| F1 Score | **1.000** | 0.976 |
| FP | **0** | 1 |
| FN | **0** | 0 |

**大逆轉**：統一 threshold 後 Mahalanobis 拿到完美分數。

### F1 Score 計算方式

```
F1 = 2 × (Precision × Recall) / (Precision + Recall)

為什麼不用普通平均？
→ 調和平均對極端值更敏感

例子：Precision=1.0, Recall=0.0
  普通平均 = (1.0 + 0.0) / 2 = 0.50  ← 看起來還好？
  調和平均 = 2×(1.0×0.0)/(1.0+0.0) = 0.00  ← 正確反映沒用
```

只要 Precision 或 Recall 任何一個很低，F1 就會被大幅拉低。

## 三、為什麼 Mahalanobis 反而贏了？

黃棋的顏色特徵太明顯：

```
黑棋：HSV H≈90-130, S低, V低（深色）
白棋：HSV H≈90,    S低, V高（淺色）
黃棋：HSV H≈25,    S=255（超鮮豔）← 和黑白棋差異極大
```

32 維 HSV 直方圖輕鬆區分，而 AutoEncoder 的 1 個誤報
來自某顆正常棋子外觀特殊（光線/遮擋），重建誤差偶然超過 threshold。

**核心規律**：越簡單的任務，統計方法反而贏；越複雜的任務，AE 更 robust。

## 四、AutoEncoder 完整原理

### 架構

```
原始 crop X（64×64×3）
         │
    ┌────▼─────────────────────────────┐
    │           AutoEncoder             │
    │                                   │
    │  [Encoder]        [Decoder]       │
    │  64×64×3          8×8×64         │
    │     ↓    壓縮         ↓  還原     │
    │  32×32×32         16×16×64       │
    │     ↓     ──────→     ↓          │
    │  16×16×64  latent  32×32×32      │
    │     ↓     8×8×64     ↓          │
    │   8×8×64           64×64×3       │
    └──────────────────────▼───────────┘
                        重建圖 X'
                           │
                    MSE(X, X')
                           │
               小 → 正常棋子 🟢
               大 → 外星人  🔴
```

### Conv2d vs ConvTranspose2d

```
Conv2d（縮小）：
  輸入（大）           輸出（小）
  ■ ■ ■ ■
  ■ ■ ■ ■    →         ■ ■
  ■ ■ ■ ■              ■ ■
  ■ ■ ■ ■
  「多合一」，用在 Encoder

ConvTranspose2d（放大）：
  輸入（小）           輸出（大）
  ■ ■       →      ■ ■ ■ ■
  ■ ■               ■ ■ ■ ■
                    ■ ■ ■ ■
                    ■ ■ ■ ■
  「一展多」，用在 Decoder
```

### Latent（潛在空間）是什麼？

```
原始 crop：  64×64×3 = 12,288 個數字
                ↓ Encoder 壓縮
   latent：   8×8×64 =  4,096 個數字（壓縮比 3x）

模型被強迫用 4,096 個數字代表整張圖
→ 必須學到「最本質的特徵」

黑棋 latent ≈「深色圓形」的編碼
白棋 latent ≈「淺色圓形」的編碼
黃棋 latent ≈「咦？沒見過，用最像的填充」→ 還原失真 → MSE 高
```

### Loss Function

```
MSE = 平均( (原圖每個像素 - 重建圖每個像素)² )

例子：
  原圖一個像素    = 0.85（亮黃色）
  重建圖同位置    = 0.20（模型只會畫黑/灰）
  誤差²          = (0.85 - 0.20)² = 0.4225

對 64×64×3 所有像素取平均 → 得到一個數字 → 就是 loss 也是異常分數
```

## 五、為什麼訓練了 AE 還需要 YOLO？

### 使用者的疑問

> 「我都訓練了 AE，為何還需要 YOLO？」

### 分工說明

```
YOLO 問的是：「棋子在圖片的哪裡？」  ← 定位問題
AE   問的是：「這顆棋子是不是外星人？」← 異常判斷問題
```

### 如果只有 AE，沒有 YOLO

```
整張 640×640 圖丟進 AE：
  黃棋只佔整張圖 1%（約 60×60 / 640×640）
  其他 99% 都能正常還原（棋盤、黑白棋）
  黃棋的 MSE 被稀釋 → 整張圖平均 MSE 仍低 → 偵測失敗！
```

**YOLO 先定位 → crop 讓黃棋佔 100% 畫面 → AE 誤差不被稀釋**

### 完整推論流程

```
棋盤圖（640×640）
      │
      ▼
┌─────────────┐
│ YOLO best.pt│  找出所有棋子位置，輸出 bounding box 座標
└─────────────┘
      │
      │  (x1, y1, x2, y2)
      ▼
我們的程式：crop = img[y1:y2, x1:x2]   ← YOLO 不截圖，我們自己截
      │
      ▼  crop（64×64×3 的小圖片）
┌─────────────────┐
│  AutoEncoder    │  Encoder 壓縮 → Decoder 還原 → 計算 MSE
│  ood_ae_model   │
└─────────────────┘
      │
      ▼
  MSE > 0.000248 → ALIEN 🔴
  MSE ≤ 0.000248 → 正常  🟢
```

## 六、YOLO 輸出的內容

### 使用者的疑問

> 「YOLO 不是只輸出 bounding box，為何你說可以截圖？crop 是什麼？」

### 釐清

```
YOLO 輸出（純數字）：
  bounding box：(x1, y1, x2, y2) = (246, 525, 306, 585)
  class：black
  conf：0.93

YOLO 不會輸出圖片。

截圖是我們的程式做的：
  crop = img[525:585, 246:306]
              ↑
    從原始圖片（NumPy array）用座標裁切

crop = 原始圖片在那個座標範圍內的像素資料
     = 一張 60×60 的小圖片
```

## 七、AE 訓練資料的來源

### 使用者確認的理解（正確）

> 「AE 只是借鑑 YOLO 的 bounding box 找到棋子位置來得到 crop，
>  做自監督學習，利用壓縮和還原來比較 X 跟 X' 的 MSE，
>  若太大就代表是外星人。」

### 訓練資料詳情

```
來源：366 張正常棋盤圖（datasets/go-games-1/train/images/）
       全部是黑/白棋，無黃棋

流程：
  366 張圖 → YOLO best.pt（conf=0.25）→ 截 crop
  → 8,408 個 crop（全是黑/白棋的截圖）
  → 餵進 AE 訓練（80 epochs）

GT：crop 本身（自監督，不需要標註）
```

### YOLO 和 AE 訓練資料的關係

```
原始標註資料
（圖片 + bounding box + class）
         │
         ├──→ 訓練 YOLO
         │    GT = bounding box + class（需要人工標註）
         │
         └──→ 用訓練好的 YOLO 截 crop
                    │
                    └──→ 訓練 AE
                         GT = crop 本身（不需要標註）
```

標註資料間接幫助了 AE（透過 YOLO 截圖），
但 AE 訓練本身是自監督的，不直接用標註。

## 八、架構根本性限制

### 使用者發現的問題

> 「OOD 方法高度依賴 YOLO，如果 YOLO 找不到，OOD 完全沒機會。
>  這是不是架構方向就錯了？」

### 白色三角形的思想實驗

```
如果外星人是「白色三角形」：

法一（Mahalanobis）：
  HSV 和白棋完全一樣（顏色相同）
  → Mahalanobis 距離 ≈ 0 → 認為是正常白棋 → FN ❌
  預估 Recall ≈ 0

法二（AutoEncoder）：
  「我見過的白棋都是圓的，三角形我不會畫」
  → 還原出圓形 → 原圖是三角形 → MSE 高 → ALIEN ✅
  但前提：YOLO 要能偵測到三角形

共同瓶頸：
  YOLO 只學過圓形 → 三角形 conf 可能 < 0.10 → 兩個方法都進不去
```

### 各架構比較

| 架構 | 依賴 YOLO？ | 適合情境 |
|------|:---------:|---------|
| 現在的架構（YOLO + OOD）| ✅ 高度依賴 | 異常物件形狀像棋子 |
| 全圖 AE | ❌ 不依賴 | 異常佔畫面較大比例（小異常易稀釋）|
| Sliding Window AE | ❌ 不依賴 | 任何異常，但計算量大 |
| PatchCore（業界主流）| ❌ 不依賴 | 工業缺陷偵測，目前最強 |

## 九、兩個方法的 Robust 比較

| 情境 | Mahalanobis | AutoEncoder |
|------|:-----------:|:-----------:|
| 顏色差異明顯（本題：黃棋）| ✅ 完美 | ✅ 好 |
| 顏色差異不明顯（深灰棋）| ❌ 容易失效 | ✅ 較撐得住 |
| 形狀異常（白色三角形）| ❌ 幾乎無效 | ✅ 有效 |
| 複雜真實場景 | ❌ 脆弱 | ✅ Robust |
| 訓練時間 | ✅ 不需要 | ❌ 需要（本次 ~50 分鐘）|

> **越複雜的任務，AutoEncoder 優勢越大；任務越簡單，統計方法反而贏。**
> 這是機器學習裡很普遍的規律。

## 十、未來提案（尚未實作）

### 提案一：白色三角形外星人實驗
- 修改 `synthesize_alien.py`：畫白色三角形取代黃色圓形
- 重跑兩個方法（AE 不需重新訓練）

### 提案二：Tesla 等自動駕駛物體偵測
- YOLO 能偵測 80 種物體（COCO），但 Tesla 需要 3D 空間感知
- 多攝影機融合 + 時序追蹤 + 深度估計
- YOLO 做得到 2D 框框，但距離/速度需要額外機制

### 提案三：業界異常偵測做法
- 工業界主流：PatchCore、PaDiM、SimpleNet（不依賴 YOLO）
- 有標籤的異常：監督式直接訓練（需要收集異常樣本）
- 無標籤異常（真正 OOD）：AE、PatchCore 等重建方法
- 現實：往往兩者並用，已知異常用監督式，未知異常用 OOD
